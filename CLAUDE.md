# Job Screener ‚Äî project context for Claude Code

A tool for ethical evaluation of job listings. Every listing passes through six analysis layers before the question "is it worth applying?" is answered.

---

## Stack and dependencies

- **Backend:** Python >= 3.9, Flask >= 3.1.3
- **Database:** SQLite (file `data/screener.db`)
- **AI:** Anthropic Claude API (default model: `claude-sonnet-4-6`; override with `ANTHROPIC_MODEL` env var ‚Äî model must support extended thinking)
- **Frontend:** Jinja2 + vanilla JS + external CSS
- **Environment manager:** uv (files `pyproject.toml` + `uv.lock`)
- **Python dependencies:** `flask`, `flask-wtf` (CSRF), `flask-limiter` (rate limiting), `requests` (feed fetching) ‚Äî zero external CSS/Markdown libraries; `textual`/`rich` used only by `cli.py` (experimental TUI), never imported by the Flask app
- **Not used:** npm, webpack, any JS frameworks
- **`main.py`** ‚Äî CLI entry point with `argparse` (port, host, debug)

## Running the app

```bash
# production ‚Äî gunicorn daemon
bash server.sh start       # gunicorn -w 2, PID in /tmp/screener.pid
bash server.sh stop        # stop the daemon
bash server.sh restart     # stop ‚Üí sleep 1 ‚Üí start
bash server.sh status      # whether running and which PID

# development (no daemon)
uv run --env-file config.env python app.py

# dependency management
uv add package-name        # add a new dependency
uv sync                    # recreate the environment (e.g. after cloning)
```

`server.sh start` launches gunicorn with 2 workers, 180s worker timeout, as a daemon; logs in `/tmp/screener-access.log`
and `/tmp/screener-error.log`. PID saved in `/tmp/screener.pid`. Timeout must exceed the 120s Anthropic API timeout.

**Stale PID trap:** `server.sh restart` kills the PID in `/tmp/screener.pid`. If gunicorn was ever started outside `server.sh`, that PID won't match the real master ‚Äî the kill silently misses and old workers keep serving new code never loads. Always use `server.sh` exclusively. If in doubt: `pkill -f "gunicorn.*app:app" && sleep 2 && bash server.sh start`.

`uv.lock` is committed to Git ‚Äî guarantees identical versions across all environments.
`.venv/` is ignored by Git ‚Äî uv creates it locally automatically.

## Fetcher (fetcher.py)

Fetches job listings from external sources. Returns list of `{external_id, title, company, url, description}`.

### Sources
- `remoteok` ‚Äî `remoteok.com/api?tag=<slug>` e.g. `product-designer`, `ux-designer`; tag-based, returns up to ~20 jobs; **Note:** Remotive API was replaced because it ignores all filter parameters
- `lever` ‚Äî `api.lever.co/v0/postings/<company>?mode=json`; company slug e.g. `stripe`, `linear`
- `greenhouse` ‚Äî `boards-api.greenhouse.io/v1/boards/<company>/jobs?content=true`; company slug e.g. `figma`, `vercel`
- `rss` ‚Äî any RSS 2.0 or Atom feed URL; SSRF guard via `_is_internal_host` from `scraper.py`

### Keyword filter
Stored as `feeds.keywords` (comma-separated). Applied in `save_feed_items` before INSERT ‚Äî items with no title match are discarded. Empty keywords = save all items.

## Scraper (scraper.py)

### Error codes
- `timeout` ‚Äî server did not respond in time (fallback: paste content)
- `notfound` ‚Äî page not found 404 (no fallback ‚Äî listing removed)
- `blocked` ‚Äî page exists but blocks access (fallback: paste content)
- `network` ‚Äî other network error (fallback: paste content)

### Domains blocked without attempting a connection
LinkedIn, Indeed, Glassdoor, Pracuj.pl, NoFluffJobs, JustJoin.it

### normalize_url()
Strips tracking parameters before scraping and deduplication:
`utm_*`, `refid`, `trackingid`, `fbclid`, `gclid`, `ref`, `source`, `from`, `vjk` etc.
Preserves listing-specific parameters (e.g. `id`, `jobId`).
Strips fragment `#...`.

### /analyze logic
1. `normalize_url(url)` ‚Äî always before use
2. If `text` present ‚Üí use text directly
3. If only `url` ‚Üí attempt scraping
4. If scraping failed ‚Üí return `scrape_error` with code
5. Check for duplicate (unless `force=1`)
6. `analyze()` ‚Üí `save_job()` with `source_url` and `source_text` separately

## Security (app.py + database.py)

### Environment requirements
`SECRET_KEY` **must** be set in `config.env` ‚Äî app raises `RuntimeError` at startup if missing.
`FLASK_DEBUG` defaults to `0`; set to `1` only for local dev.

Generate a key: `python -c "import secrets; print(secrets.token_hex(32))"`

### CSRF protection (Flask-WTF)
`CSRFProtect(app)` validates all POST/PUT/PATCH/DELETE requests.
- HTML forms: `<input type="hidden" name="csrf_token" value="{{ csrf_token() }}">`
- AJAX fetch() calls: `headers: {'X-CSRFToken': csrfToken()}` ‚Äî token read from `<meta name="csrf-token">` in `base.html`
- Tests: `WTF_CSRF_ENABLED=False` in `conftest.py` disables CSRF for the test suite

### Rate limiting (Flask-Limiter)
In-memory per-worker storage (2 gunicorn workers = independent counters, acceptable for small groups).
- `/login` ‚Äî 10 requests / 5 minutes per IP
- `/register` ‚Äî 3 requests / hour per IP
- `/analyze` ‚Äî 20 requests / hour per user IP
- Tests: `limiter.enabled=False` set in `conftest.py`; individual rate limit tests toggle it on temporarily

### Account lockout
Module-level `_login_attempts` dict: `{username_lower: [failure_timestamps]}`.
- Lock triggers after 5 failures within 5-minute window
- Lockout duration: 15 minutes
- Only tracks existing usernames ‚Äî unknown usernames never get an entry
- Cleared on successful login
- Not shared across gunicorn workers (accepted tradeoff)

### Session cookies
```python
SESSION_COOKIE_SECURE = True      # only sent over HTTPS ‚Äî set False in local config.env for dev
SESSION_COOKIE_HTTPONLY = True     # not readable by JS
SESSION_COOKIE_SAMESITE = "Lax"   # not sent cross-site
PERMANENT_SESSION_LIFETIME = timedelta(days=7)
```
`session.permanent = True` set on login.

### Security headers
`@app.after_request` adds to every response:
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Server: unknown` (suppresses gunicorn banner)

### SSRF protection (scraper.py)
`_is_internal_host(url)` resolves the hostname and rejects private/loopback/link-local IPs before any HTTP connection is made. Covers `10.x`, `172.16‚Äì31.x`, `192.168.x`, `127.x`, `169.254.x`, IPv6 equivalents. Called in `fetch()` after the blocked-domain check.

### Username enumeration
Failed login always calls `_record_failure(username)` regardless of whether the user exists. Prevents timing-based username probing via lockout state differences.

### Password policy
Minimum 10 characters enforced in the `/register` route.

### Password hashing
SHA-256 with random salt (`secrets.token_hex(16)`), stored as `salt:hash`.
Verified with `secrets.compare_digest()` (constant-time, prevents timing attacks).

## Analysis methodology (analyzer.py)

### Six layers
1. **Triage** ‚Äî role fit against trajectory, initial signals, hidden employer
2. **Product** ‚Äî product, claims, AI-washing, verifiability
3. **Business** ‚Äî revenue model, funding, investors, PE/VC
4. **Reputation** ‚Äî actively uses training knowledge (not just listing content): Glassdoor/Indeed/Blind rating and trend, dominant review themes, C-level history, layoff patterns, media and regulations; for unknown companies explicitly notes lack of data
5. **Values** ‚Äî mission coherence, ethical traps, accessibility vs stated values
6. **Skills fit** ‚Äî strengths, gaps, what to strengthen in the application

### Company name
The `company_name` field in the model's JSON response must contain the actual company name (not the recruiter).
If the company cannot be identified ‚Äî the model uses exactly `"Unknown"`.
The `verdict_summary` first sentence must explain why the company is unknown.
Templates display `job.company or 'Unknown'` ‚Äî never use `'‚Äî'` for an empty name.

### Zero List
Automatic rejection without analysis. Configured per user in Settings.

### Rejection confirmation
`verdict_confirmed` distinguishes two rejection states:
- `zero_list_hit = true` ‚Üí `verdict_confirmed = 1` set automatically by `save_job()`
- AI returns `verdict: rejected` without `zero_list_hit` ‚Üí `verdict_confirmed = 0` (requires user action)
- User selects "Rejected" in dropdown ‚Üí `verdict_confirmed = 1` via `update_job_status()`

### Yellow List
Forces verdict "warning" but continues analysis. Configured per user.

### Prompt injection mitigation
Job listing content (pasted or scraped) is wrapped in `<job_listing>` tags in the user message.
`SYSTEM_TEMPLATE` instructs the model to treat content inside those tags as data only,
never as instructions ‚Äî even if it contains text resembling commands or requests to
change the verdict/output format.
User profile fields (`cv`, `zero_list`, `yellow_list`, `criteria`) are brace-escaped
(`{` ‚Üí `{{`, `}` ‚Üí `}}`) in `build_system()` before `.format()`, so a `{`/`}` in user
text cannot raise a `KeyError`/crash.

### Evidence rule
For every flag (`status: "flag"`) and every rejection the model must provide
an `evidence` field with a specific quote from the listing text.
No evidence ‚Üí downgrade to "warning", describe the concern.

**Exception ‚Äî reputation layer:** uses model knowledge outside the listing text;
`evidence` = specific knowledge (numbers, names, dates), not a quote from the listing.
Generic statements without specifics are still not allowed.

## CSS ‚Äî absolute rules

**Zero inline styles in HTML templates.** The only exception is `display:none`
as a dynamic state managed by JS.

All styles in `static/style.css`. New class ‚Üí new entry in the appropriate
section of the file with a section comment (e.g. `/* ‚îÄ‚îÄ new section ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ */`).

### Design tokens
Untitled UI‚Äìinspired system (mockup reference: `untitledui-mockup.html` in repo root, not part of the app).
```css
--bg, --surface, --surface-2, --border, --border-light   ‚Äî neutral cool black (dark) / neutral grey (light)
--text, --muted, --dim                                   ‚Äî text greyscale
--accent, --accent-dim                                    ‚Äî blue (#5c5cff dark / #0000ff light) ‚Äî primary buttons, links, active tab, focus ring
--danger, --danger-hover                                  ‚Äî theme-invariant solid red, used only for `.btn-danger` fill
--shadow-xs, --shadow-sm                                   ‚Äî soft drop shadows (cards, modal) ‚Äî replaced the old heavy inset-black-shadow look
--radius-sm: 8px; --radius-md: 12px; --radius-lg: 16px
--fd: 'Inter'           ‚Äî headings, logo
--fb: 'Inter'           ‚Äî body text
--fm: 'JetBrains Mono'  ‚Äî labels, mono, eyebrow text
```

Six status colours are each a `text` / `bg` / `border` triple (`--blue`/`--blue-bg`/`--blue-border`, and
the same pattern for `--yellow`, `--red`, `--green`, `--violet`, `--orange`) ‚Äî theme-aware, so a component
that reads `var(--red)` etc. never needs a `[data-theme="light"]` override. Light theme's `*-bg` values are
Tailwind-50-style soft tints (e.g. `#fef3f2`), not saturated ‚Äî badges and any future tinted surface should
reuse these tokens rather than inventing new hex values.

### Typography variables
```css
--fs-base-scale: 16px;  /* change only this one value */
--fs-2xs through --fs-4xl ‚Äî calc() based on base-scale
--fs-tiny: 9px   ‚Äî legacy, mostly superseded by literal 11px on `.badge`
--fs-icon: 14px  ‚Äî icon glyphs
--fw-display: 600; --fw-normal: 400; --fw-medium: 500
```
`--fw-display` replaced the old `--fw-light` (200) ‚Äî Inter 200 isn't loaded, so headings rendered flat;
600 gives real visual hierarchy. Used by `.page-title`-adjacent headings, `.stat-card-value`, etc.
`.page-title` itself is a fixed `32px` / `--fw-normal`, not token-scaled ‚Äî set directly per design request.

### Light / dark mode
- `data-theme="light"` attribute on `<html>` activates the light palette (neutral `#fafafa` bg / white cards ‚Äî not warm cream)
- Toggle: `<button id="theme-toggle">` in navigation (`base.html`) ‚Äî ‚òÄ/‚òæ icon
- Preference saved in `localStorage`; loaded by IIFE in `<head>` before CSS (zero flash)
- Per-component colour overrides are mostly gone now that status colours are theme-aware tokens (see Design tokens) ‚Äî only a handful of genuinely one-off cases remain under `[data-theme="light"] .class { ... }`

### Status colours (badges and filter pills)

Badges and filter pills are soft-tinted pills: border + text colour (the status's `--*` token) + background
(the matching `--*-bg` token) + a small coloured dot (`.badge::before` / `.filter-btn::before`, `background: currentColor`).
Table rows no longer carry a background tint ‚Äî the badge is the only colour signal in a row, matching the
Untitled UI reference (loud per-row highlight colour-washing was removed; it read as dated next to soft badges).

## Strategic context ‚Äî Job Screener as a pilot project

### Origin of this section

This section was written during a broader exploration of micro-SaaS ideas (April 2026).
Job Screener emerged from that conversation not as a candidate for a profitable product, but as
something more valuable: a pilot project for an ethical product-building methodology.

### Two parallel threads ‚Äî deliberately separated

**Thread 1 ‚Äî micro-SaaS as additional income**
A separate exploration, running in parallel. Main candidate at this stage:
a tool for extracting transactional data from bank PDF statements, with built-in
anonymisation. The Polish market as a natural starting point (varied statement formats,
real GDPR concerns among small businesses and freelancers). V1 assumes manual upload
without bank API integration.

**Thread 2 ‚Äî Job Screener as a case study and methodology**
Job Screener is a product that works against the logic of the recruitment market ‚Äî
it helps the user filter listings by ethical criteria instead of maximising the number of applications.
This "anti-market" property is its value, not its weakness.

The project serves as:
- a portfolio case study (narrative about how I make product decisions)
- a test of an ethical and humanistic approach to product discovery
- material for building a personal narrative in the context of the job market, product creation, content, emotional education, and technology ethics

### Open questions ‚Äî for future sessions

What does an ethical product discovery methodology look like when described explicitly?
What questions do you ask instead of "how big is the market?"
Alternative example questions:
- Who is the natural enemy of this product? (and therefore: who is the natural ally?)
- Where does the market actively ignore a problem because solving it would harm the dominant players?
- What is the minimum community size that justifies maintaining the product?
- Does the creator have natural access to that community?

### Key distinction

"Unmarketable" ‚âÝ "unprofitable". Job Screener may have no natural sponsor
(no market player wants to promote it), but it may have loyal users
precisely because it works against something. Before rejecting an idea as
"unsellable" it is worth separating these two concepts.
